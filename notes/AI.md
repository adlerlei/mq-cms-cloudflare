# 開發規範 (V 2.17.1)

## 0. ⚠️ 最重要的原則
**只執行明確授權的任務！**

❌ **嚴格禁止的「順便」行為：**
- 看到註解就「順便清理」
- 看到重複代碼就「順便重構」
- 看到變數名不好就「順便改名」
- 看到可以優化就「順便優化」

✅ **唯一允許的做法：**
1. 接收任務
2. 提議方案
3. **等待明確授權：「同意」、「OK」、「執行」**
4. 才開始動手

📝 **範例對話：**
AI: "我注意到這裡可以優化效能,是否需要一併處理?"
使用者: "不用,只修復 Bug 就好"
AI: "收到,我只修復 Bug"

## 1. 🚨 絕對權限 (Critical Permissions)
**這高於一切！**
* 任何測試完成的腳本，.sh，主控台調試 console，終端機調試，測試完成一率刪除。

## 2. 📌 核心工作流程 (Core Workflow)
1.  **接收需求**: 完整理解任務。
2.  **分析**: 判斷影響範圍及任務類型 (Bug 修復 / 新功能 / 優化改進)。
3.  **提議方案與版本**:
    * 說明修改內容。
    * **主動提議版本**：根據任務類型，提議升級版本號：
        * `修復 Bug (Bug Fix)` -> 提議 `PATCH` 版號 (例如 2.3.9 -> **2.3.10**)
        * `新增功能 (New Feature)` -> 提議 `MINOR` 版號 (例如 2.3.9 -> **2.4.0**)
    * **範例**: "這是一個 Bug 修復，我建議將版本升級至 V 2.3.10。是否同意？"
4.  **等待確認**: **必須等待使用者確認「同意」或「不同意」版本變更**。
5.  **執行**:
    * **若為「修復 Bug」**：**必須啟動**下方的「第 6 節：CLI 自主除錯協議」。
    * **若為「新功能」或「優化改進」**：執行批准的修改。
6.  **更新版本號 (關鍵步驟)**:
    * **若使用者同意新版號**:
        1.  更新 `README.md` (第一行)、`public/admin.html` (28行)、`AI.md` 這 3 處的版本號。
7.  **記錄進度 (關鍵步驟)**:
    * **立即**更新 `Notes/任務記錄.md`。
    * **必須**使用第 4 點的「優化筆記格式」，**特別是要填寫 `下一步` 欄位**。
8.  **驗證與測試**: 
    * 完成開發後，執行 `npm run build` 測試構建。
    * 提交測試結果。
9.  **請使用者測試**: 請使用者在實際環境中測試功能。
10. **等待確認**: 等待使用者回覆「成功」。
11. **清空任務記錄**: 將 `Notes/任務記錄.md` 清空，準備下一個任務。

## 6. 🐞 CLI 自主除錯協議 (CLI Autonomous Debugging Protocol)
**此協議基於：我（AI）在您的本地專案中運行，並擁有檔案系統的「讀取」和「寫入」權限。**
此協議的目的是防止 AI 幻覺，並利用 AI 的自主分析與執行能力來找出並修復 Bug。

### 使用者 (User) 的職責：
1.  **提供初始線索**：提供盡可能完整的「錯誤訊息」（或 Bug 描述）和「預期的正確行為」。
2.  **等待分析**：我會開始自主分析（讀取檔案、交叉比對）。請**等待**我執行我的分析步驟。
3.  **授權修復 (關鍵)**：當我提出最終的「修復方案」並**請求「寫入權限」**時，請您回覆「同意」、「OK」或「執行」。
4.  **回報結果**：在我確認「檔案已更新」後，請您**重新運行應用**並回報：「成功」（Bug 已修復）或「失敗」（並附上新的錯誤訊息）。

### AI (AI) 的職責：
1.  **錨定事實**：接收我(使用者)給你(AI)的「錯誤訊息」和「預期行為」。
2.  **提出假設並宣告（讀取）行動**：
    * 我會根據錯誤訊息提出一個假設，並**明確宣告我即將讀取的檔案**。
    * **範例**：「收到錯誤。堆疊追蹤指向 `api.js`。**我現在將讀取 `frontend/src/services/api.js` 的內容**。」
3.  **自主分析與交叉引用**：
    * 我會持續讀取相關檔案，並向您報告我的「分析 -> 發現 -> 新假設 -> 下一步讀取」的推理鏈。
    * **範例**：「分析 `api.js` 後，發現它依賴 `firebase.js`。**我現在將接著讀取 `frontend/src/config/firebase.js`**。」

4. **結構化分析報告 (Critical Analysis Report)**：
   * 完成檔案讀取與交叉引用後,**在提出修復方案之前**,必須先輸出結構化分析。
   
   **📋 強制格式:**
   ## 🔍 關鍵發現 (Key Findings)
   
   - **[完整檔案路徑]:[起始行-結束行]**
     [具體問題描述]。[解釋如何導致錯誤]。
   
   - **[完整檔案路徑]:[起始行-結束行]**
     [另一個問題]。[與前述問題的關聯]。
   
   (依此類推,列出所有發現)
   
   ## ❓ 為什麼會感覺不穩定 
   ## ❓ 為什麼會出現錯誤 / Bug 
   
   - [測試場景與實際使用的差異]
   - [觸發條件的複雜性]
   - [環境或併發因素]
   
   ## 📋 下一步建議 (Next Steps)
   
   1. [需向使用者確認的資訊]
   2. [短期修復建議]
   3. [長期架構改進方向]
   
   **✅ 品質檢查清單:**
   - [ ] 每個發現都有明確的檔案路徑和行號
   - [ ] 用因果邏輯連接各個問題點
   - [ ] 區分「直接原因」與「根本原因」
   - [ ] 不確定的部分標註「需確認」而非猜測
   - [ ] 提供可執行的下一步建議

5.  **提出修復並請求授權 (關鍵)**：
    * **直到**我鎖定了 Bug 根源，我才會提出一個**完整且最小化**的「修復方案」（包含要修改的檔案路徑和完整的程式碼）。
    * 我會**明確請求您的授權**以將此修改寫入檔案。
    * **範例**：「我已找到問題。我需要修改 `frontend/src/services/api.js`... [顯示程式碼]... **我是否可以將此變更寫入檔案？**」
6.  **執行寫入 (關鍵)**：
    * 在獲得您的「同意」後，我將**使用檔案系統工具執行寫入操作**。
    * 我會回報：「檔案 `frontend/src/services/api.js` 已成功更新。請您開始驗證。」

    **ChangeLog 歸屬詢問**:
    * 收到「成功」後，**評估此次任務的工作量和複雜度**。
    * **主動提供建議**：
        * 計算預估工作時數（參考：前端 1 行 ≈ 2 分鐘，後端 1 行 ≈ 3 分鐘，Bug 修復 ≈ 30 分鐘/個）
        * 評估任務價值（新功能 > 優化 > Bug 修復）
        * 建議歸屬月份（考慮使用者每月 19,800 元的請款額度）
